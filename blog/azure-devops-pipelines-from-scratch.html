Setting Up Azure DevOps Pipelines from Scratch

If your team is still building and deploying manually, you're wasting hours every week and introducing risk with every release. Here's how to set up Azure DevOps pipelines that automate the entire process.

Why YAML Pipelines?

Azure DevOps supports both the classic GUI editor and YAML pipelines. Use YAML. It lives in your repository alongside your code, so it's version controlled, reviewable in pull requests, and portable. The classic editor is fine for learning, but YAML is the standard for production.

Build Pipeline Basics

A basic .NET build pipeline does four things: restore NuGet packages, build the solution, run tests, and publish the output as an artifact. In YAML, this takes about 30 lines. Trigger it on every push to your main branch and on pull requests.

Key tip: use dotnet test --logger trx to generate test results that Azure DevOps can display natively in the pipeline summary. Add code coverage with coverlet and publish the results — your team will actually look at coverage when it's visible in every PR.

Release Pipeline

For deployment, I use multi-stage YAML pipelines. Stage 1: build and test. Stage 2: deploy to staging with automatic approval. Stage 3: deploy to production with manual approval gate. Each stage references the artifact from Stage 1, ensuring you deploy exactly what you tested.

Infrastructure as Code

Don't create Azure resources manually. Use Terraform or Bicep in your pipeline to create and update infrastructure. This means your staging and production environments are identical, new environments can be spun up in minutes, and infrastructure changes go through the same review process as code changes.

Essential Additions

Once the basics are working, add: branch policies requiring successful builds before merging, SonarQube or similar for code quality gates, Slack/Teams notifications for build failures, automatic dependency updates with Dependabot, and scheduled builds for security scanning.

Getting Started

If this feels overwhelming, start simple. Get a build pipeline running that restores, builds, and tests on every push. That alone catches most bugs before they reach production. Add deployment automation later. The perfect pipeline is the enemy of the good pipeline.
