Modernise Legacy Apps Without a Full Rewrite

The temptation when facing a legacy codebase is to scrap it and start fresh. Don't. Full rewrites are the number one cause of failed software projects. Here's how to modernise incrementally, reducing risk while delivering value continuously.

Why Full Rewrites Fail

The legacy system has years of bug fixes, edge case handling, and domain knowledge baked in. A rewrite throws all of this away. Meanwhile, the old system still needs maintaining, so you're paying for two systems at once. And the rewrite always takes longer than estimated — typically 2-3x longer — because the complexity that made the old system messy doesn't disappear in the new one.

The Strangler Fig Pattern

Named after tropical fig trees that gradually envelop their host, this pattern replaces legacy components one at a time. New functionality is built in the modern stack. Old functionality is migrated piece by piece. A routing layer directs traffic to either the old or new system. Eventually, the old system has nothing left and can be switched off.

Where to Start

Don't start with the most complex component. Start with something small, well-understood, and low-risk. A settings page. A simple report. A lookup API. This lets you establish the new architecture, set up CI/CD, and prove the approach works — all without risking anything critical.

The API Boundary

Put an API layer between the old and new systems. The new frontend calls the API. The API talks to both old and new backends depending on which component has been migrated. This gives you a clean separation point and lets you migrate at whatever pace works for your team.

Dealing With the Database

The database is usually the hardest part. Legacy systems often have tightly coupled stored procedures, triggers, and direct table access from multiple applications. My approach: keep the existing database initially and build new services that read from it. Gradually introduce new tables/schemas for migrated components. Eventually consolidate when the old system is fully replaced.

Measuring Progress

Track what percentage of traffic goes through the new system versus the old. This gives you a clear, objective measure of migration progress that stakeholders can understand. Set realistic milestones — 25%, 50%, 75% — and celebrate each one.
